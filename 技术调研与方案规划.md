# EasyCampus 安卓签到管理应用 - 技术调研与方案规划

## 一、项目概述

### 1.1 项目背景
开发一款功能完善的安卓签到管理应用，集成多平台签到功能，包括课堂派、雨课堂、长江雨课堂及畅课等主流教学平台，并提供课程表查看功能。

### 1.2 核心功能需求
1. **多平台签到管理**
   - 课堂派 (ketangpai.com)
   - 雨课堂 (yuketang.cn)
   - 长江雨课堂 (changjiang.yuketang.cn)
   - 畅课 (changke)

2. **课程表查看功能**
   - 参考开源项目 GUET课表 的实现方案
   - 支持周视图/日视图切换
   - 课程提醒功能

3. **用户系统**
   - 多平台账号管理
   - 本地数据加密存储
   - 自动登录与Token管理

## 二、技术选型报告

### 2.1 开发语言与框架

#### 2.1.1 开发语言：Kotlin
**选择理由：**
- Google官方推荐的首选语言，所有新API优先支持Kotlin
- 与Java 100%互操作，可无缝集成现有库
- 空安全特性减少崩溃率
- 协程支持简化异步编程
- 2025年Android开发生态已全面转向Kotlin优先

#### 2.1.2 UI框架：Jetpack Compose
**选择理由：**
- Google官方现代声明式UI工具包
- 开发效率比传统XML提升50%以上
- 响应式UI更新，自动处理状态变化
- 与Kotlin深度集成，支持协程    
- 已成为2024-2025年Android开发的主流选择
- Material Design 3组件开箱即用

**对比传统XML方案：**
| 特性 | Jetpack Compose | XML + View |
|------|-----------------|------------|
| 开发效率 | 高（代码即UI） | 低（分离布局） |
| 性能 | 智能重组优化 | 手动优化 |
| 学习成本 | 中等 | 低 |
| 生态支持 | 官方主推 | 维护模式 |
| 动态UI | 原生支持 | 需额外处理 |

### 2.2 架构模式：MVVM + Clean Architecture

#### 2.2.1 架构分层
```
Presentation Layer (UI Layer)
├── UI (Jetpack Compose Screens)
├── ViewModel (StateFlow/SharedFlow)
└── UI State Management

Domain Layer (Business Logic)
├── Use Cases (Interactors)
├── Repository Interfaces
└── Domain Models

Data Layer (Data Access)
├── Repository Implementations
├── Data Sources (Local/Remote)
├── API Services
└── Database (Room)
```

#### 2.2.2 选择理由
- **关注点分离**：UI、业务逻辑、数据访问完全解耦
- **可测试性**：各层可独立单元测试
- **可维护性**：代码结构清晰，易于扩展
- **Google官方推荐**：与Jetpack组件完美配合

### 2.3 核心组件选型

#### 2.3.1 依赖注入：Hilt
**选择理由：**
- Google官方推荐的依赖注入框架
- 基于Dagger但大幅简化配置
- 与Jetpack组件深度集成
- 编译时依赖图验证，运行时零开销
- 支持ViewModel、WorkManager等组件注入

#### 2.3.2 异步编程：Kotlin Coroutines + Flow
**选择理由：**
- 轻量级线程，比RxJava更易上手
- Flow提供响应式数据流处理能力
- StateFlow替代LiveData，无生命周期问题
- 结构化并发，自动取消机制

**Flow类型选择：**
- **StateFlow**：用于UI状态管理（如登录状态、课程列表）
- **SharedFlow**：用于一次性事件（如Toast提示、导航事件）
- **Flow**：用于数据流（如网络请求、数据库查询）

#### 2.3.3 网络请求：Retrofit + OkHttp
**选择理由：**
- 行业标准REST客户端
- 支持协程suspend函数
- OkHttp提供强大的拦截器机制
- 支持自动JSON解析（Gson/Moshi）

**优化策略：**
- 连接池复用
- 请求/响应缓存
- 超时配置优化
- 重试机制

#### 2.3.4 本地存储：Room + DataStore

**Room（结构化数据）：**
- 类型安全的SQL封装
- 编译时SQL验证
- 支持协程和Flow响应式查询
- 数据库迁移支持

**DataStore（键值对偏好设置）：**
- 替代SharedPreferences
- 基于协程和Flow
- 类型安全（Preferences DataStore / Proto DataStore）
- 事务性存储保证一致性

#### 2.3.5 后台任务：WorkManager
**选择理由：**
- 适配Doze模式和App Standby
- 支持周期性任务（签到检查）
- 保证任务执行（即使应用关闭）
- 支持约束条件（网络、电量等）

### 2.4 安全方案

#### 2.4.1 数据加密
- **Android Keystore System**：存储加密密钥
- **AES-256加密**：敏感数据（密码、Token）
- **EncryptedSharedPreferences**：替代普通SharedPreferences
- **SQLCipher**：数据库加密（可选）

#### 2.4.2 网络安全
- **Certificate Pinning**：防止中间人攻击
- **HTTPS强制**：所有API请求
- **Token管理**：JWT + Refresh Token机制
- **Root检测**：检测设备是否被Root

### 2.5 性能优化策略

#### 2.5.1 UI渲染优化
- **LazyColumn/LazyRow**：大数据集列表优化
- **remember/derivedStateOf**：减少不必要的重组
- **@Stable/@Immutable注解**：帮助Compose识别稳定性

#### 2.5.2 网络优化
- **请求合并**：批量处理签到检查
- **智能重试**：指数退避算法
- **离线优先**：本地缓存，网络恢复后同步

#### 2.5.3 启动优化
- **懒加载**：非必要组件延迟初始化
- **Splash Screen API**：规范启动体验

## 七、技术风险与应对

### 7.1 平台API变动风险
**风险：** 第三方平台API可能随时变动
**应对：** 
- 设计灵活的平台适配器架构
- 实现API版本管理机制
- 建立快速响应机制

### 7.2 登录验证风险
**风险：** 平台可能增加验证码、滑块验证等
**应对：**
- 引导用户手动获取Cookie
- 实现WebView辅助登录
- 提供手动签到备选方案

### 7.3 安全性风险
**风险：** 用户账号信息泄露
**应对：**
- 本地数据加密存储
- HTTPS强制传输
- 密钥托管在Android Keystore

### 7.4 后台执行限制
**风险：** Android系统对后台任务限制
**应对：**
- 使用WorkManager适配系统策略
- 引导用户设置电池优化白名单
- 前台服务保活（必要时）

## 十、方案确认记录

经过讨论，最终确定以下方案：

### 10.1 架构方案
| 项目 | 选择 |
|------|------|
| 服务端架构 | **纯客户端方案** - 所有数据本地存储，无需服务器 |
| 自动签到 | **半自动模式** - 检测到签到时推送通知，用户确认后执行 |
| 平台支持 | **全平台并行** - 同时开发所有平台支持 |

### 10.2 安全方案
| 项目 | 选择 |
|------|------|
| 凭证存储 | **本地加密存储** - Android Keystore + AES-256加密 |
| 生物识别 | 可选支持 |
| PIN码保护 | 可选支持 |

### 10.3 功能方案
| 项目 | 选择 |
|------|------|
| 课表导入 | 教务系统导入 + 手动添加编辑 |
| 课程提醒 | 支持，可自定义提前时间 |
| 桌面小部件 | 支持今日课程Widget |

### 10.4 项目配置
| 项目 | 选择 |
|------|------|
